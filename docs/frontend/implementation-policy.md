# 프론트엔드 구현 정책

이 문서는 백엔드 API가 완성되지 않은 상태에서도 프론트 화면 개발을 진행하기 위한 기준입니다.

## 기본 방향
- 현재 프론트엔드는 목업 화면을 뼈대로 사용한다.
- API가 이미 존재하는 화면은 실제 API 연동까지 진행한다.
- API가 아직 없거나 명세가 부족한 화면은 목업 데이터와 TODO 주석으로 먼저 구현한다.
- 백엔드 구현이 완료되면 `docs/frontend/api-contract.md` 기준으로 목업을 API로 교체한다.

## 기능별 개발 방식

| 영역 | 현재 방침 | Codex 작업 기준 |
|---|---|---|
| 인증 | 권한별 로그인 화면 분리 | USER/MERCHANT/ADMIN 흐름을 분리해서 구현 |
| 회원가입 | 자동 로그인 우선 | 가능하면 회원가입 성공 후 로그인 처리, 불가능하면 로그인 화면 이동 |
| 장소/지도 | 실제 API 연동 가능 | 주변 장소/상세/길찾기 우선 연동 |
| 검색 | 목업 우선 | 화면과 UX 먼저 구성, API 연결 TODO 남김 |
| 축제 | 목업 우선 | 카드/상세/캘린더 UI 먼저 구성 |
| 커뮤니티 | 목업 우선 | 게시글/댓글/작성 UI 먼저 구성 |
| 찜하기 | MVP 제외 | 버튼/상태만 구성, 실제 저장은 TODO |
| QR 결제 | 목업 우선 | 사용자 플로우에 맞게 화면 구현, 실제 결제 API는 TODO |
| PG 충전 | PG URL 없이 구현 | 충전 요청/완료 화면 중심으로 구현 |
| 지도 SDK | 일단 보류 | 지도 SDK 없이 리스트/카드/외부 길찾기 링크 중심 구현 |

## API가 없는 기능의 구현 규칙
- `mockData.js` 또는 화면 내부 fixture로 데이터 구성
- 추후 교체할 API endpoint 후보를 TODO로 남김
- 실제 서버 호출처럼 보이는 가짜 delay는 과하게 넣지 않음
- 사용자에게는 “준비 중”보다 자연스러운 빈 상태/목업 상태 UI 제공

## MVP 제외 기능
- 장소 찜하기 저장
- 실제 QR 결제 처리
- PG 결제창 연동
- 고도화된 지도 SDK 인터랙션

## 우선 개발 추천 순서
1. 권한별 인증 화면
2. 홈 화면
3. 장소 목록/상세/길찾기
4. 엽전 잔액/충전 화면
5. QR 결제 목업 플로우
6. 검색/축제/커뮤니티 목업 화면
7. 상인/관리자 화면
